# 官方文档与外部证据闸门

这个闸门用于减少第三方用法脑补，同时避免每次任务都过度查文档。

## 何时必须运行

满足任一条件时，必须先做官方文档与外部证据核实：

- 新增、升级、配置或明显改变第三方框架、插件、SDK、CLI、API、云服务、数据库驱动、认证、支付、部署平台或硬件集成。
- 计划依赖当前外部事实，例如 API 限制、版本支持、平台策略、价格、发布状态或官方迁移要求。
- 仓库里没有同版本、同 API surface 的可靠本地模式。
- 你不确定方法名、配置键、CLI 参数、生命周期钩子、初始化顺序、迁移步骤或兼容性行为。

## 何时允许跳过

只有以下条件全部满足时，才允许跳过新的外部检索：

- 仓库附近已经有同版本、同 API surface 的工作代码。
- 本次实现沿用该本地模式，没有改变集成边界或配置语义。
- 设计文档已经记录了该版本、该用法对应的证据条目。
- 没有测试失败、Opus 阻断意见或本地类型/源码证据与现有记录冲突。

## 证据记录格式

把证据写入设计文档的 `External Evidence and Official Docs` 表格：

| Dependency/Surface | Version/Target | Claim or usage rule | Official source | Checked date | Applies to plan step | Stale trigger | Confidence |
| --- | --- | --- | --- | --- | --- | --- | --- |
| | | | | | | | |

建议规则：

- `Version/Target` 写准确版本、目标平台或 API 版本。
- `Claim or usage rule` 只写本次实现真正依赖的约束。
- `Official source` 优先写官方文档、官方仓库、官方 release notes、官方 migration guide 或官方 API reference。
- `Stale trigger` 写明什么变化会让这条证据失效，例如“升级到 React 20”“切换到新网关”“Gemini API 版本变化”。

## 查询路由

优先按这个顺序获取证据：

1. 仓库内现有实现、测试、类型定义、锁文件、配置文件、安装产物、CLI `--help` 输出。
2. 官方文档、官方仓库、官方 API reference、官方 release notes、官方 migration guide。
3. 维护者源码、类型定义、schema、官方示例仓库。
4. 社区帖子只作为线索，不作为最终依据；引用前必须被官方或维护者证据覆盖。

## 使用原则

- 先明确问题，再查文档。例如先问“这个 CLI 在 v3 下如何注册 plugin”，而不是笼统地“去看看文档”。
- 先确认目标版本，再读对应文档，不要混用不同版本的说明。
- 只读回答当前问题所需的最小章节。
- 一旦当前 step 的用法契约、关键参数和验证路径已经明确，就停止继续扩展阅读。
- 实现过程中，如果出现新 API surface、版本漂移、测试与文档冲突或 Opus 质疑，再补查对应章节。

## 禁止事项

- 不要脑补第三方 option、flag、config key、hook、endpoint 或 import path。
- 不要把社区帖子、博客或问答站当作最终计划证据。
- 不要把私有控制台页面、账号信息、密钥或生产数据贴进设计文档。
- 无法验证且高风险的外部行为，不要继续推进实现；应记录不确定性并转为阻塞报告。
